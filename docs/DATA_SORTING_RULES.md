# 데이터 정렬 규칙

이 문서는 화면에 노출되는 데이터의 정렬 기준을 관리하는 기준 문서다.
정렬 로직을 변경할 때는 구현 코드와 이 문서를 같은 변경에서 함께 수정한다.

## 공통 원칙

1. 검색과 카테고리 필터를 적용한 뒤 페이지네이션으로 데이터를 자른다.
2. 날짜 목록은 별도 규칙이 없으면 최신순(내림차순)으로 정렬한다.
3. 정렬 값이 같으면 기존 입력 순서를 유지한다. 임의로 제목이나 ID를 추가
   정렬 기준으로 사용하지 않는다.
4. 날짜가 없거나 해석할 수 없는 항목은 유효한 날짜가 있는 항목보다 뒤에 둔다.
5. 이 문서에 정렬 기준이 없는 목록은 원본 JSON 또는 Notion 응답 순서를 유지한다.

## 날짜 해석

공통 날짜 정렬은 `src/lib/utils.ts`의 `parseDate`와 `sortByDateDesc`를 기준으로 한다.

- 허용 형식: `YYYY`, `YYYY.MM`, `YYYY.MM.DD`, `YYYY-MM-DD`, `YYYY/MM/DD`
- 월이 없으면 1월, 일이 없으면 1일로 해석한다.
- `현재`는 정렬 시점의 오늘 날짜로 해석한다.
- `periods`가 있는 데이터는 각 기간의 시작일 중 가장 최근 날짜를 대표 날짜로 사용한다.
- 공통 정렬 함수는 원본 배열을 변경하지 않고 복사본을 반환한다.

## 화면별 정렬 규칙

### Home — 최근 등록된 기록

홈 중앙 카드 그리드는 게시일 최신순으로 정렬하고 선택한 카테고리의 상위
3개를 표시한다. 댓글과 조회수는 카드 하단에 표시하지만 최근 등록 순서에는
영향을 주지 않는다.

`일지` 필터는 하나의 물리 데이터 키가 아니라 아래 Notion 콘텐츠를 합친 가상
카테고리다.

1. `education`: 교육일지
2. `blog`: 일지

두 데이터를 합친 뒤 게시일 최신순으로 정렬한다.

### Home — 인기 콘텐츠

대상: 홈 오른쪽 `인기 콘텐츠` 목록

우선순위:

1. Giscus/GitHub Discussions 댓글 수 내림차순
2. Google Analytics 페이지 조회수 내림차순
3. 게시일 최신순

전체 카테고리 기준 상위 4개를 표시한다. 댓글과 조회수가 모두 0이면 게시일
최신순으로 결정된다.

- 구현: `src/app/page.tsx`의 `compareByEngagement`
- 통계 생성: `scripts/engagement/fetch.mjs`
- 통계 저장: `src/data/indexes/engagement.json`

### Home — 인기 태그

1. 전체 Devlog 글에서 태그별 사용 횟수를 계산한다.
2. 사용 횟수 내림차순으로 정렬한다.
3. 사이드바에는 상위 12개를 표시한다.
4. 횟수가 같으면 원본 글과 태그를 순회하며 처음 발견된 순서를 유지한다.
5. 태그가 12개를 넘으면 `…` 버튼을 표시하고, 팝업에는 정렬된 전체 태그를
   표시한다.

### Home — 카테고리

아래 고정 순서를 사용한다.

1. 기술 학습
2. 문제 해결
3. 대회·행사
4. 일지(`education` + `blog`)

### Devlog — 탭과 글 목록

탭은 아래 고정 순서를 사용한다.

1. 전체 글
2. 기술 학습 기록
3. 문제 해결 기록
4. 대회/행사
5. 일지

개별 탭과 일지 하위 목록은 게시일 최신순이다. `전체 글` 탭은 단순한 전체
날짜 혼합이 아니라 다음 그룹 순서를 사용한다.

1. 기술 학습·문제 해결·대회/행사 글을 최신순으로 표시
2. 교육일지·일지 글을 최신순으로 표시

따라서 일지 글이 더 최신이어도 일반 Devlog 글 그룹 뒤에 표시된다. 검색과
패키지 필터를 적용한 결과를 페이지당 6개씩 표시한다.

### Devlog — 패키지와 일지 카테고리

1. `전체 보기`는 항상 첫 번째다.
2. 나머지는 표시 이름 기준으로 정렬한다.
3. 영문으로 시작하는 이름을 한글 이름보다 먼저 둔다.
4. 같은 문자권에서는 한국어 `Intl.Collator`의 숫자 인식·대소문자 무시 규칙을
   사용한다.

### 교육일지

교육일 최신순으로 정렬한 뒤 검색 결과를 페이지당 9개씩 표시한다. Notion
동기화 파일 자체는 `lastEditedTime`이 있으면 최종 수정일, 없으면 교육일을
기준으로 최신순 저장하지만, 화면에서는 교육일을 기준으로 다시 정렬한다.

### Devlog 흐름 캐러셀 컴포넌트

`RecentDevlogFlowSection`을 화면에 연결할 경우 섹션은 아래 고정 순서를 사용하고
한 화면에 카테고리 카드 3개씩 표시한다.

1. 기술 학습 기록
2. 문제 해결 기록
3. 대회/행사
4. 교육일지
5. 일지

각 섹션 내부 글은 게시일 최신순이다.

### Projects

프로젝트의 `periods`에 포함된 시작일 중 가장 최근 날짜를 기준으로 최신순
정렬한다. 참여 작업/개인 작업 탭과 검색 필터를 적용한 뒤 페이지당 6개씩
표시한다.

About 화면의 프로젝트 쇼케이스는 별도 정렬을 수행하지 않는다. 원본
`projects.json` 순서를 유지하면서 참여 작업을 먼저, 개인 작업을 다음 그룹으로
나눈다.

### Career와 기타 정적 목록

경력, 학력, 자격증, About 하이라이트처럼 별도 정렬 함수가 없는 목록은 원본
JSON 배열 순서를 그대로 사용한다. 표시 순서를 변경하려면 원본 데이터의
배열 순서를 변경한다.

## 동기화 파일 저장 순서

- Notion 일지: 게시일 최신순으로 `devlog.json`에 저장
- Notion 교육일지: 최종 수정일(`lastEditedTime`) 우선, 없으면 교육일 기준 최신순
- 참여도 통계: pathname 오름차순으로 JSON 키를 저장

통계 JSON의 pathname 정렬은 diff를 안정적으로 유지하기 위한 저장 규칙이며,
화면의 인기 순위와는 관계없다.

## 1. 입력하신 내용 기준 핵심 정리

### 기본 코드 작성의 목적 재확인
자바의 가장 기본이 되는 getter, setter, 그리고 생성자(Constructor)를 왜 생성해야 하는지 그 본질적인 이유를 다시 한번 되짚어보는 계기가 되었습니다.

### Lombok 라이브러리와 Builder 패턴의 도입
예전에는 모든 코드를 일일이 직접 작성(손코딩)하곤 했지만, 이제는 Lombok 라이브러리를 활용하여 소스 코드의 가독성과 관리 용이성을 한층 높일 수 있게 되었습니다. 특히 관리해야 할 변수가 많은 복잡한 객체를 생성할 때는 생성자 오버로딩 대신 빌더(Builder) 패턴을 적극적으로 활용함으로써, 반복적인 코드를 줄이고 개발 생산성을 크게 향상시키는 방법을 확인했습니다.

### Inner Class 활용과 독립 클래스 분리 전략
초기 코드 작성 단계에서는 하나의 파일 안에 내부 클래스(Inner Class) 형태로 먼저 구현하여 빠르게 테스트를 진행한 뒤, 검증이 완료되면 별도의 독립된 클래스 파일로 분리하여 선언하는 방식을 취했습니다. 이 과정을 통해 코드의 재사용성을 높일 수 있었을 뿐만 아니라, 무분별하게 static class를 남발하지 않도록 제어하여 메모리 효율성까지 함께 고려해 볼 수 있었습니다.

## 2. 추가 보충: 개발자로서 한 걸음 더 나아가기

작성해주신 구조적인 접근법에 몇 가지 기술적인 디테일을 더하면, 실무에서 더욱 견고하고 효율적인 코드를 작성하는 데 도움이 됩니다.

### 💡 무분별한 Setter 지양하기
Lombok이 제공하는 `@Data`나 `@Setter`를 사용하면 코드는 편해지지만, 객체의 값이 어디서나 변경될 수 있어 객체의 일관성(Invariance)이 깨질 위험이 있습니다.

- 값이 변하지 않는 객체는 필드를 `final`로 선언하고 `@Getter`와 `@Builder`만 조합하여 불변 객체(Immutable Object)로 만드는 것이 안전합니다.
- 값의 변경이 꼭 필요하다면 `setAge()` 같은 단순 setter보다는 `changeAddress()`처럼 의도가 명확히 드러나는 비즈니스 메서드를 별도로 정의하는 것이 유지보수에 유리합니다.

### 💡 내부 클래스(Inner Class) 분리 시 메모리 누수(Memory Leak) 방지
처음 테스트 단계에서 클래스 내부에 서브 클래스를 둘 때, `static`을 붙이지 않은 일반 내부 클래스는 외부 클래스에 대한 참조를 숨은 채로 유지하게 됩니다.

이 상태로 내부 클래스가 어디선가 계속 참조되고 있다면, 외부 클래스가 더 이상 쓰이지 않더라도 가비지 컬렉터(GC)가 메모리에서 해제하지 못하는 메모리 누수가 발생할 수 있습니다.

따라서 완전히 독립된 파일로 분리하기 전, 파일 하나에서 임시로 내부 클래스를 쓰실 때는 가급적 `static class`로 선언하여 외부 클래스와의 연결고리를 끊어주는 것이 메모리 관리에 더 안전합니다.

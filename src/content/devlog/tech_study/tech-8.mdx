내가 설정한 환경과 애플리케이션을 그대로 묶어 배포하기 위한 Docker Image 생성 방법을 두 가지 관점에서 정리했습니다.

---

## 1. 기존 컨테이너를 활용한 추출 (Commit & Push)

<Indent>
실행 중인 베이스 컨테이너에 직접 쉘 접속하여 프로그램 설치 및 환경 설정을 마치고, 그 상태 그대로 '스냅샷'을 떠서 이미지를 만드는 직관적인 방식입니다.

**작업 과정:**
1.  `docker exec -it [컨테이너명] /bin/bash` 로 접속하여 필요한 작업 수행
2.  `docker commit [컨테이너명] my_custom_image:1.0` (이미지 굽기)
3.  `docker tag my_custom_image:1.0 my_registry/my_custom_image:1.0` (태그 설정)
4.  `docker push my_registry/my_custom_image:1.0` (원격 저장소에 업로드)

**단점:** 
*   이미지가 어떻게 구성되었는지 이력을 추적할 수 없어 유지보수가 매우 어렵습니다. 실무 환경보다는 일회성 테스트용으로 적합합니다.
</Indent>

## 2. Dockerfile에 직접 작성하여 생성 (IaC 방식)

<Indent>
'Infrastructure as Code' 철학에 맞춰 이미지를 조립하는 설명서(Dockerfile)를 코드로 작성하고 버전 관리를 하는 방식입니다.

### 2-1. `entry.sh` 없이 단순 명령어 기반 생성
대부분의 표준적인 애플리케이션에 사용되는 방식입니다.
```dockerfile
FROM ubuntu:22.04
# 기본 패키지 설치
RUN apt-get update && apt-get install -y nginx
# 로컬 소스 복사
COPY src/ /var/www/html/
# 포트 노출 선언
EXPOSE 80
# 컨테이너 실행 시 데몬 모드가 아닌 포그라운드 모드로 엔진 가동
CMD ["nginx", "-g", "daemon off;"]
```

### 2-2. 복잡한 `entry.sh`를 활용한 고난도 커스텀 이미지
컨테이너 기동 시점에 DB 연결 대기, 초기 마이그레이션 스크립트 실행, 동적 환경변수에 따른 Config 파일 템플릿 렌더링 등 복잡한 '전처리' 과정이 필요할 때는 쉘 스크립트를 진입점(Entrypoint)으로 삼습니다.

```dockerfile
FROM openjdk:17-slim
WORKDIR /app
COPY myapp.jar /app.jar
# 전처리 쉘 스크립트 복사 및 실행 권한 부여
COPY entry.sh /entry.sh
RUN chmod +x /entry.sh

# 컨테이너가 시작될 때 무조건 스크립트 실행
ENTRYPOINT ["/entry.sh"]
```

**`entry.sh` 예시 로직:**
```bash
#!/bin/bash
echo "1. 데이터베이스 연결을 대기합니다..."
# (DB 연결 확인 로직)

echo "2. 최신 설정 정보를 환경변수로부터 적용합니다..."
# (sed, awk 등을 활용한 config 치환 로직)

echo "3. 메인 자바 애플리케이션을 실행합니다."
# exec 명령을 사용하여 쉘 프로세스를 자바 프로세스로 교체(PID 1 유지)
exec java -jar /app.jar
```
</Indent>

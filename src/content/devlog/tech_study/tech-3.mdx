컨테이너 기술은 현대 클라우드 네이티브 아키텍처의 핵심 요소로 자리 잡았습니다. 이 문서는 Docker의 기본 개념과 구동되는 내부 프로세스를 다룹니다.

---

## 1. 컨테이너 기술의 핵심 개념

<Indent>
Docker는 하이퍼바이저 기반의 하드웨어 가상화와 달리, 리눅스 커널의 기능을 활용한 **운영체제 레벨 가상화**를 제공합니다.

*   **Namespace:** 파일 시스템, 프로세스 트리, 네트워크 인터페이스 등 시스템 자원을 프로세스별로 격리하는 기능입니다. 이를 통해 각 컨테이너는 독립된 운영체제를 가진 것처럼 동작합니다.
*   **Cgroups (Control Groups):** CPU, 메모리, 디스크 I/O 등의 하드웨어 리소스 사용량을 그룹별로 할당하고 제한하는 기능입니다. 한 컨테이너가 시스템 전체 자원을 독점하는 것을 방지합니다.
</Indent>

---

## 2. Docker Architecture

<Indent>
Docker는 클라이언트-서버(Client-Server) 아키텍처를 따릅니다.

1.  **Docker Client:** 사용자가 CLI 또는 API를 통해 Docker 데몬에게 명령을 전달하는 창구입니다. (`docker run`, `docker pull` 등)
2.  **Docker Daemon (dockerd):** 클라이언트의 API 요청을 수신하여 이미지 빌드, 컨테이너 실행 및 관리 등 실제적인 작업을 수행하는 백그라운드 서비스입니다.
3.  **Docker Registry:** Docker Image들이 저장되는 원격 또는 로컬 저장소입니다. (예: Docker Hub)
</Indent>

---

## 3. Container 구동 프로세스

<Indent>
사용자가 `docker run` 명령어를 입력했을 때 시스템 내부에서 일어나는 과정은 다음과 같습니다.

1.  **명령 수신:** 클라이언트가 `docker run <image_name>` 명령을 Docker Daemon에 전달합니다.
2.  **이미지 확인 및 Pull:** Daemon은 지정된 이미지가 로컬 캐시에 존재하는지 확인합니다. 없다면 Registry에서 이미지를 다운로드(pull)합니다.
3.  **컨테이너 생성 (Create):** 다운로드된 이미지를 기반으로 읽기 전용(Read-Only) 파일 시스템 레이어를 구성합니다.
4.  **읽기/쓰기 레이어 결합:** Union File System (UFS)을 사용하여 읽기 전용 레이어 위에 쓰기 가능한(Writable) 임시 레이어를 얹어 컨테이너의 최종 파일 시스템을 구성합니다.
5.  **격리 환경 할당:** Namespace와 Cgroups를 통해 네트워크 인터페이스, IP 주소, 리소스 제한 등을 설정합니다.
6.  **프로세스 실행 (Start):** Dockerfile에 정의된 기본 명령어(`CMD` 또는 `ENTRYPOINT`)를 컨테이너 내부에서 실행하여 컨테이너 구동을 완료합니다.
</Indent>

WAS 서버 이중화 세션 클러스터링을 구축할 때, 인프라 네트워크 연결만으로는 완벽한 로드 밸런싱(LB) 및 데이터 공유가 이루어지지 않습니다. WAS 내부 애플리케이션(Java) 단에서 데이터를 자체 검증하거나 암호화하는 별도의 키가 일치하지 않는다면, 정상적인 데이터 공유가 불가능합니다.

입력하신 네트워크 설정을 기준으로 현상을 정리하고, Java 내부에서 발생할 수 있는 검증 이슈와 해결 방법을 분석해 보겠습니다.

## 1. Tomcat 네트워크 클러스터링(Tribes) 설정 분석

<Indent>
현재 설정은 Tomcat의 Tribes 멤버십 및 통신 레이어 설정입니다.

*   **`<Membership>` (McastService):** `228.0.0.4:45564` 멀티캐스트 주소를 통해 서버들이 서로의 존재를 확인(Heartbeat)합니다. 500ms 주기로 신호를 보내고, 3000ms 동안 신호가 없으면 멤버에서 제외합니다.
*   **`<Receiver>` (NioReceiver):** 다른 톰캣 서버가 보낸 세션 데이터를 받기 위해 4000번 포트를 열어두는 비동기(NIO) 수신 채널입니다.

**💡 요약:** 이 설정은 서버 'A'와 서버 'B'가 서로 이야기를 나눌 수 있는 '전화선'을 물리적으로 연결한 상태입니다. 하지만 통화 내용(데이터 포맷)을 서로 이해하고 신뢰할 수 있는지는 별개의 소프트웨어적인 문제입니다.
</Indent>

## 2. Java 내부 검증 키 및 공유 실패 원인

<Indent>
네트워크 연결이 완벽해도 Java 애플리케이션 수준에서 다음과 같은 '검증 키'나 조건이 맞지 않으면 세션 공유 시 에러가 발생하거나 데이터가 유실됩니다.

### ① serialVersionUID (직렬화 버전 ID) 미일치
Java 객체를 네트워크로 전송하려면 **직렬화(Serializable)** 과정을 거쳐야 합니다. 데이터를 보내는 톰캣의 클래스 버전과 받는 톰캣의 클래스 버전이 다르면 `InvalidClassException`이 발생합니다.
*   **원인:** 소스 코드가 조금만 바뀌어도 Java가 자동으로 생성하는 `serialVersionUID`가 달라지기 때문입니다.
*   **해결 방안:** 세션에 담기는 모든 VO/DTO 모델 클래스에 명시적으로 UID 값을 고정해야 합니다.

```java
private static final long serialVersionUID = 1L;
```

### ② Spring Security 등의 CSRF 토큰 및 암호화 키 불일치
애플리케이션 내부에 세션 데이터를 암호화하는 Local Key가 존재할 수 있습니다.
*   **원인:** JWT나 Session 기반 보안 설정을 할 때, 서버 내부 메모리의 Random Seed로 암호화 키를 생성하도록 해두면 서버 A와 서버 B의 키가 서로 달라집니다. A가 암호화해서 보낸 세션을 B가 풀지 못해 강제로 무효화(Invalidate) 처리를 해버립니다.
*   **해결 방안:** 암호화에 사용되는 비밀키(Secret Key)를 톰캣 환경변수나 외부 설정 파일(`application.properties` 등)을 통해 두 서버가 동일하게 맞추어야 합니다.

### ③ 외부 세션 저장소 이용 시 Credential 미일치
만약 톰캣 자체 클러스터링(DeltaManager)이 아니라, Redis나 Hazelcast 같은 별도 솔루션을 Java 소스 내(Spring Session 등)에서 연결해 쓰고 있다면, 해당 저장소에 접근하는 인증 키(Password/Token)가 맞지 않아 공유에 실패할 수 있습니다.
</Indent>

## 3. 안정적인 클러스터링을 위한 체크리스트

<Indent>
Tomcat 클러스터링 네트워크 설정 후에도 Java 단에서 정상적으로 세션이 공유되는지 확인하려면 다음 항목들을 반드시 점검해 보세요.

*   [ ] 세션에 담기는 모든 객체(VO, DTO, Entity)가 `implements Serializable`을 명시적으로 구현했는가?
*   [ ] 클러스터링 대상 서버들의 배포 파일(WAR, JAR) 내부 클래스들(특히 `serialVersionUID`)이 완전히 동일하게 컴파일되었는가?
*   [ ] 암호화 및 보안 관련 라이브러리가 각 서버 고유의 MAC 주소나 시스템 정보를 기반으로 동적인 키를 생성하고 있지는 않은가? (고정 키 사용 권장)
</Indent>
